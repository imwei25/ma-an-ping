# 技术路线组织逻辑范式

> 基于16本NSFC成功申请书的逻辑提炼，重点总结"为什么这样设计技术路线"，而非固定模板。

## 一、研究内容→技术路线的构建逻辑

### 核心构建链

```
研究目标 → 研究内容 → 关键科学问题 → 研究方法 → 技术路线
```

每一步的构建逻辑：

1. **研究目标→研究内容**：目标分解为可操作的研究任务
2. **研究内容→关键科学问题**：提炼核心科学难点
3. **关键科学问题→研究方法**：针对难点选择方法
4. **研究方法→技术路线**：将方法嵌入可执行的研究流程

### 构建原则

- 技术路线必须响应研究目标
- 技术路线必须支撑研究内容
- 技术路线必须体现研究方法
- 技术路线必须形成研究闭环
- 技术路线图与正文必须对应

## 二、六种项目类型的典型技术路线

### 类型1：数据驱动型

```
数据预处理 → 特征提取/选择 → 模型构建 → 实验验证 → 应用分析
```

逻辑：从数据到知识，从模型到验证

路线图特点：
- 总体图强调数据流向
- 分内容图按"数据→方法→模型→验证"展开
- 每个分内容图包含具体的算法/模型名称

### 类型2：预测分析型

```
特征工程 → 预测建模 → 效果验证 → 应用部署
```

逻辑：从特征到预测，从验证到应用


路线图特点：
- 强调多阶段递进（异常探测→影响因素→监测预警）
- 每阶段有独立的技术路线图
- 体现"先过滤后深化"的逻辑

### 类型3：优化决策型

```
问题建模 → 求解方法 → 方案优化 → 实证验证
```

逻辑：从问题到方案，从优化到验证

路线图特点：
- 总体图强调"评估→优化→验证"闭环
- 分内容图按专题展开
- 每个专题有明确的输入输出

### 类型4：方法研究型

```
理论研究 → 方法研究 → 实践应用
```

逻辑：从理论到方法，从方法到应用


路线图特点：
- 强调递进关系（理论→方法→应用）
- 方法创新点在路线图中突出标注
- 应用部分体现方法的实际价值

### 类型5：实证研究型

```
文献分析 → 数据收集 → 实证检验 → 政策建议
```

逻辑：从文献到数据，从实证到建议


路线图特点：
- 强调数据收集的多样性（问卷/访谈/爬虫）
- 实证检验部分包含多种分析方法
- 政策建议部分体现研究的实践价值

### 类型6：机制研究型（重点项目）

```
理论构建 → 机理揭示 → 动态模拟 → 配置优化 → 应用验证
```

逻辑：从理论到机制，从模拟到优化，从优化到应用


路线图特点：
- 总体图体现全链条逻辑
- 每个课题有独立的技术路线图
- 课题之间有明确的输入输出关系
- 体现"评估→预测→配置→保障"闭环

## 三、技术路线展开层级范式

### 展开层级选择逻辑

**默认采用细粒度展开**（按孙研究展开），与研究内容的三级结构严格对齐。

仅在以下情况使用粗粒度展开：
- 用户显式指定"按粗粒度写""整体写一段"
- 用户提供的材料本身就是粗粒度组织的

### 细粒度展开范式（默认）

技术路线按**子研究→孙研究**展开，每个子研究包含：帽段 + 路线图 + 孙研究独立段落。

#### 细粒度展开的文字结构

```
[子研究帽段：4步逻辑]
  1. 数据/平台基础
  2. 研究方法概述
  3. 研究思路概括
  4. 引出孙研究

[子研究路线图]

（1）[孙研究1]
  1. 输入/前置条件
  2. 问题定位
  3. 方法设计
  4. 处理流程（首先→其次→然后→最后）
  5. 输出/结果
  6. 衔接/价值

（2）[孙研究2]
  1. 输入/前置条件（承接孙研究1的输出）
  2. 问题定位
  3. 方法设计
  4. 处理流程
  5. 输出/结果
  6. 衔接/价值

（3）[孙研究3]
  ...
```

#### 细粒度展开的路线图结构

子研究路线图应体现孙研究的递进关系：

```
输入数据
  ↓
孙研究1 → 输出1
  ↓
孙研究2 → 输出2
  ↓
孙研究3 → 输出3
  ↓
验证/应用
```

每个孙研究节点标注使用的方法。

#### 孙研究间的衔接逻辑

孙研究之间的衔接是细粒度展开的关键，常见衔接模式：

| 衔接模式 | 说明 | 典型表述 |
|---|---|---|
| 数据流衔接 | 孙研究1的输出数据作为孙研究2的输入 | "基于(1)中提取的特征..." |
| 模型流衔接 | 孙研究1构建的模型作为孙研究2的基础 | "在(1)构建的XXX模型基础上..." |
| 结果流衔接 | 孙研究1的分析结果指导孙研究2的设计 | "根据(1)的分析结果..." |
| 并行衔接 | 孙研究之间独立，但共享输入 | "与(1)并行，基于同一数据集..." |

### 粗粒度展开范式（仅在用户指定时使用）

技术路线按**子研究**展开，每个子研究写1段合并方案+1张图。

#### 粗粒度展开的文字结构

```
[子研究1名称]。[输入/前置条件]，首先，[步骤1]；其次，[步骤2]；
然后，[步骤3]；最后，[步骤4]。[衔接下一子研究]。
本专题研究方案的技术路线如图X所示。

[子研究1路线图]

[子研究2名称]。[输入/前置条件]，首先，[步骤1]；其次，[步骤2]；
然后，[步骤3]；最后，[步骤4]。[衔接下一子研究]。
本专题研究方案的技术路线如图X所示。

[子研究2路线图]
  ...
```

#### 粗粒度展开的路线图结构

子研究路线图将孙研究内容合并在流程中：

```
输入数据
  ↓
步骤1（对应孙研究1内容）→ 中间结果1
  ↓
步骤2（对应孙研究2内容）→ 中间结果2
  ↓
步骤3（对应孙研究3内容）→ 最终结果
  ↓
验证/应用
```

## 四、技术路线图设计方案

### 总体路线图设计

**核心要素**：
1. 项目名称（根节点）
2. 研究内容/课题（二级节点）
3. 子研究内容（三级节点）
4. 研究方法（叶节点或侧边标注）
5. 逻辑关系（箭头）

**设计要点**：
- 研究内容按逻辑顺序排列（不是简单并列）
- 体现"基础→核心→应用"的递进关系
- 标注关键方法
- 体现闭环（反馈箭头）

### 分内容路线图设计

**核心要素**：
1. 输入数据/前置条件
2. 处理步骤（按序排列）
3. 使用的方法（标注在步骤旁）
4. 输出结果
5. 验证方式

**设计要点**：
- 每个步骤有明确的输入输出
- 方法标注清晰
- 体现"数据→处理→结果→验证"的完整流程
- 与总体路线图对应

## 四、模块划分规律

### 按研究内容划分（最常见）

每个研究内容对应一个技术路线模块，模块之间有逻辑衔接。

### 按研究阶段划分

按"基础研究→核心研究→应用研究"划分，每个阶段包含多个研究内容。

### 按课题划分（重点项目）

每个课题对应一个技术路线模块，课题之间有输入输出关系。

## 五、输入输出组织规律

### 模块间输入输出

```
模块1输出 → 模块2输入
模块2输出 → 模块3输入
模块3输出 → 反馈验证 → 模块1优化
```

### 典型输入输出链

```
数据 → 特征 → 模型 → 结果 → 验证 → 优化 → 应用
```

### 闭环设计

所有项目的技术路线都应形成闭环：
- 正向：数据→方法→结果
- 反向：验证→优化→改进

## 六、方法嵌入规律

### 嵌入原则

1. 方法不是独立存在的，而是嵌入在研究流程中
2. 每个方法出现在解决特定问题的位置
3. 方法之间有输入输出关系
4. 同一方法可能出现在多个阶段

### 嵌入方式

1. 在路线图中标注方法名称
2. 在文字描述中展开方法细节
3. 通过衔接句连接方法部分和路线部分

## 七、反馈闭环设计规律

### 单闭环

```
研究内容1 → 研究内容2 → 研究内容3 → 验证 → 反馈优化
```

### 双闭环（重点项目）

```
外环：理论→方法→应用→反馈→理论
内环：数据→模型→验证→优化→数据
```

### 闭环表述

在可行性分析中体现闭环：
```
这构成了一个循序渐进、由理论到应用的科学研究过程。
```
